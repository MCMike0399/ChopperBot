This repo directory at `/home/burbujamc/Documentos/ChopperBot` **IS** that live deployment. **Edits here affect the running bot ‚Äî but only after `pnpm run build`**, since systemd runs `node dist/index.js` directly (not `pnpm`, not `tsx watch`). To pick up changes:

```bash

systemctl --user status chopperbot.service        # confirm: active (running)
```

The single unit is **`chopperbot.service`** (a `systemd --user` unit at `~/.config/systemd/user/chopperbot.service`). It runs `node dist/index.js` with `WorkingDirectory=` the repo root (so dotenv reads `.env` from here), `Restart=always`/`RestartSec=15`, and a crashloop guard (`StartLimitIntervalSec=300`/`StartLimitBurst=10`). `loginctl enable-linger` is set so it starts at **boot** without an interactive login. Running `node` directly (no shell rc sourced) is deliberate ‚Äî it stops a stale `export ACCESS_KEY_ID=...` in `~/.bashrc` from shadowing `.env`.

The unit is generated from the template at **`deploy/systemd/chopperbot.service`** by substituting two placeholders (`__REPO__` ‚Üí repo root, `__NODE__` ‚Üí node path). That snapshot is not auto-synced ‚Äî if you change the installed unit, re-copy it into `deploy/` (and run `systemctl --user daemon-reload`).

**The three auxiliary launchd agents were intentionally NOT ported.** On the Mac, `com.user.chopperbot-watcher` (log-watcher ‚Üí desktop notifications), `com.user.chopperbot-daily-summary` (21:00 notification), and `com.user.chopperbot-health-check` (every 30 min) all fired `osascript`/Sosumi notifications, and `chopperbot-status.sh` was a SwiftBar menu-bar plugin ‚Äî none of that exists on a headless Pi. **On the Pi there are no timers and no log-watcher: the bot alerts the config Discord channel directly, and everything else is `journalctl`** (see "Logs & observability"). The alert surface (all through the shared `sendAdminAlert` in `src/discord/admin-alert.ts`, all Spanish, all swallow their own send errors):
- **IG monitor** (from the scheduler): auth-expired, circuit-breaker, budget-exhausted, polling-resumed, and the 21:00 daily status digest ‚Äî see "Instagram monitor scheduler". These replaced the Mac watcher's role and revived the old `chopperbot-daily-summary`.
- **LLM health** (`src/llm/health.ts`, added 2026-06-12): every Bedrock Converse call in `ask()` reports to `llmHealth`; deterministic 4xx (400/401/403/404/422 ‚Äî config/protocol errors that never self-heal: a `ValidationException`, revoked/insufficient IAM creds, a missing `bedrock:InvokeModel` permission, a bad model id) alert on the FIRST failure, transient errors (Throttling/5xx/network) after 3 consecutive, rate-limited to 1 per 6 h, with a one-shot recovery notice (`‚úÖ LLM recuperado`) when service returns. Status lives on `err.$metadata.httpStatusCode` (AWS SDK), with a name-based fallback (`Throttling*` transient, `Validation/AccessDenied/ResourceNotFound*` deterministic). Sink injected in `app.ts` post-login; without a sink (tests/scripts) it just logs (`llm.health.alerting` / `llm.health.recovered`). Verify end-to-end: `npx tsx scripts/verify-llm-alert.ts` (posts a clearly-marked synthetic alert).
- **Crash restarts** (`src/lifecycle.ts`): SIGINT/SIGTERM write `data/.shutdown-clean`; boot consumes it and compares against `data/.boot-stamp`. Booted-before + no clean marker = the previous process died (crash/OOM/SIGKILL) and systemd revived it ‚Üí `‚öÝÔ∏è ChopperBot se reinici√≥ tras un fallo` after login. Crashloop-debounced (previous boot <15 min ago ‚Üí log `lifecycle.unclean_restart_detected` only, no Discord spam; systemd's StartLimitBurst caps the loop anyway). Note this means a `kill -9` test WILL page the channel.
If broader push alerts are ever wanted (phone push without Discord), the cleanest add is still a log-tail service POSTing to ntfy.sh (`deploy/bin/chopperbot-log-watcher.py` has the matching logic; only the `notify()` sink changes).

The old macOS artifacts are kept in **`deploy/`** purely for reference/rollback: `deploy/launchd/` = the 4 original plists, `deploy/bin/` = the 6 helper scripts. They are **not** used by the Pi.

## Architecture

**One Discord channel = at most one specialized Capability**, with `general_chat` as the baseline fallback for any unbound guild channel. A Capability is a self-contained bundle of system prompt + tools + private SQLite namespace. Four ship today:

- `configuration` ‚Äî admin console, hardcoded to one channel.
- `calendar` ‚Äî **global** (one shared server calendar, like instagram_monitor; rewritten from the old per-user model 2026-06-21). Mods manage events in natural language from the bound INPUT channel; every create/update/delete is persisted, rendered into the month's PDF template, and published (with a master ICS) to a separate OUTPUT channel. Daily/weekly/monthly recurrence. See "Calendar capability (global)".
- `instagram_monitor` ‚Äî global account list + fan-out: one row per IG username, posts pushed to **every channel currently bound to the capability**. Per-channel dedup table gives newly-bound channels no backfill.
- `general_chat` ‚Äî baseline conversation/redirect. Never bound to a channel; runs automatically when the bot is @-mentioned in any guild channel that has no specialized binding.

### Boot sequence (`src/app.ts`)

1. Open shared SQLite at `data/chopperbot.db` (WAL, FK on).
2. Migrate the framework `users` directory (reserved capability id `__framework__`).
3. Instantiate the candidate capabilities, call `init()` on each **before Discord login**. A capability that throws is skipped; the bot continues with the rest.
4. Load channel‚Üícapability bindings: env-var seed ‚Üí DB (via `ConfigurationCapability.bootStore()`) ‚Üí build router.
5. Construct the Discord client, register handlers, `client.login()`.
6. Call each capability's optional `start()` for background work (instagram_monitor's scheduler starts here).

**Lazy deps gotcha:** `CapabilityInitDeps` exposes `getDiscordClient`, `getRegistry`, `getRouter`, `getUserDirectory` as **getters that throw if called during `init()`**. They are only safe to invoke at `buildTurn()` time (post-bootstrap). This is by design ‚Äî capabilities that need them (configuration) capture the getters in `init()` and call them later.

### Per-turn flow (`src/discord/handlers.ts` ‚Üí `src/llm/client.ts`)

1. Message arrives. Bot responds only if @-mentioned or a reply to one of its own messages in an authorized channel.
2. `userDirectory.upsert()` registers the Discord user (lazy, idempotent ‚Äî bumps `last_seen_at`).
3. `buildHistory()` walks the reply chain backward (max 8 turns / 16k chars), strips the `_‚Ä¶sigue ‚Üì_` continuation footer from bot turns, and reverses to chronological order.
4. `capability.buildTurn(ctx)` returns `{ system, tools }` for *this* message. Capabilities decide whether to rebuild every turn (calendar embeds an upcoming-events snapshot, configuration embeds current time) or cache.
5. `ask({ system, messages, tools })` drives a multi-turn agent loop against Bedrock (max `MAX_TOOL_ITERATIONS`). Within a single `ask()` call, identical `(toolName, JSON-stable-input)` tool calls are deduplicated from a per-turn cache (only successful results are cached).
6. If the loop runs out of iterations while still emitting tool_calls, a final **forcing pass** runs the model **without tools** to extract a text answer instead of leaving the user with a fallback string.
7. Long replies are split by `chunkBotReply()` ‚Äî markdown code fences are preserved across chunk boundaries (closed with ``` and reopened with the same language tag); non-tail chunks get the `_‚Ä¶sigue ‚Üì_` footer so users know to reply to the last message.

### LLM client (`src/llm/client.ts`)

Amazon Bedrock via the **Converse API** (`@aws-sdk/client-bedrock-runtime`, `ConverseCommand`). The Converse API gives one uniform messages/tools/image interface across every Bedrock model, so swapping `BEDROCK_MODEL_ID` needs no code change. Migrated off Moonshot Kimi Code (OpenAI SDK) on 2026-06-23. Non-obvious behaviors:

- **Effort tiers (added 2026-06-23).** `ask()` takes an `effort?: 'high' | 'medium' | 'low'` (default `'high'`) that picks the model: `high` ‚Üí `BEDROCK_MODEL_ID` (default Sonnet 4.6), `medium` ‚Üí `BEDROCK_MODEL_MEDIUM` (default Haiku 4.5), `low` ‚Üí `BEDROCK_MODEL_LOW` (default `us.amazon.nova-pro-v1:0`). Resolved by `modelForEffort()` in `client.ts`; the chosen `model`+`effort` ride the `agent_turn` log line. **All three must be image-capable** (the IG path needs vision). Usage today: **chat + calendar = high** (Sonnet's reliable multi-turn tool-calling ‚Äî see the bake-off below), **IG classifier = medium** (Haiku). `low` is wired but currently **unused** (reserved for a cheap bulk-vision pre-pass). Tiers were picked by **`scripts/effort-experiment.ts`** (2026-06-23): a synthesized dense-Spanish "report this group" flyer (the moderator's exact failure case) + a replay of **real DB captions** + a casual-chat tone check. Findings: **vision text-reading is NOT a differentiator** ‚Äî Sonnet, Haiku, Nova Pro AND Nova Lite all read the embedded group name + platform off the flyer perfectly. So the recurring "the bot can't see the image" complaint was **never a model gap** ‚Äî the classifier was simply **never handed the image** (caption-only by design, now fixed; see the IG classifier note). Haiku ‚âà Nova Pro on caption-relevance agreement (both 21/24 vs the stored Sonnet labels, both clean, zero parse failures), but on **chat tone** Sonnet/Haiku are warm + grounded while Nova Pro/Lite drift into generic fairy-tale prose ‚Äî that's why the moderator-facing classifier summary uses **Haiku, not Nova**. The user's first instinct ("low reads the image, medium writes the post") collapsed to a **single medium call with the image attached**: one LLM call, the medium model both reads the flyer and writes the summary (no two-stage pipeline, no doubled per-post cost).
- **Model id is `BEDROCK_MODEL_ID`** (default `us.anthropic.claude-sonnet-4-6` ‚Äî Claude Sonnet 4.6 via the **us cross-region inference profile**). Chosen after a full bake-off (2026-06-23, `scripts/model-competition.ts` + `scripts/calendar-bedrock-smoke.ts` replaying a real mod conversation). Findings, worst‚Üíbest: **Amazon Nova Pro/Lite** corrupt recurring edits + leak `<thinking>` into replies; **Llama 4 Maverick/Scout** leak raw Python code / fail the vision gate; **Qwen3-VL-235B** (`qwen.qwen3-vl-235b-a22b`) is the best image-capable **open-weight** model ‚Äî clean replies + vision ‚Äî but fragments/duplicates on multi-turn edits; **Haiku 4.5** (`us.anthropic.claude-haiku-4-5-20251001-v1:0`) is a clean, cheaper (~$1/$5) middle with ~50% reliability on the hardest edits; **Sonnet 4.6** gave the cleanest real-conversation results (right weekday, one-off w/ location, duplicate detection, no fragmentation) ‚Äî closest to the retired Kimi. MiniMax M2 / DeepSeek are **text-only** (fail the IG vision requirement). All shortlisted models are multimodal + tool use; switch with no code change. The **`us.` prefix is the cross-region inference profile** required in us-east-1; **non-Amazon models (Claude/Qwen/etc.) need model access enabled for the account** (Nova is on by default). After changing the model, re-run `npx tsx scripts/live-bedrock-smoke.ts` (+ `scripts/calendar-bedrock-smoke.ts` for the calendar). The IAM user `chopperbot-readonly` carries an inline `chopperbot-bedrock-invoke` policy granting `bedrock:InvokeModel` on all `foundation-model/*` + `inference-profile/*` (broadened during the bake-off).
- **Auth is an IAM access-key pair**: `ACCESS_KEY_ID` + `SECRET_ACCESS_KEY` (+ optional `AWS_SESSION_TOKEN`), region `AWS_REGION` (default us-east-1). The var names are deliberately **un-prefixed** (not `AWS_ACCESS_KEY_ID`) so they don't collide with an ambient AWS CLI credential on the host. The IAM principal needs `bedrock:InvokeModel` (and `InvokeModelWithResponseStream`) on both the foundation-model ARNs **and** the inference-profile ARN ‚Äî a cross-region profile routes to us-east-1/us-east-2/us-west-2, so the foundation-model resource must be region-wildcarded. The live deployment's `chopperbot-readonly` user carries an inline `chopperbot-bedrock-invoke` policy covering both `amazon.nova-*` and `anthropic.claude-*` foundation models + their `us.`/`global.` inference profiles (added 2026-06-23). A missing permission surfaces as `AccessDeniedException` (403) ‚Äî `classifyLlmError` treats it as deterministic and pages the config channel on the first failure.
- **No `temperature` sent.** The client passes only `inferenceConfig.maxTokens` (= `MAX_OUTPUT_TOKENS`); sampling params are left at the model default.
- **Tool loop contract.** On `stopReason === 'tool_use'`, the assistant turn (with its `toolUse` blocks) is echoed back into history verbatim ‚Äî Converse requires the originating `toolUse` block be present for the matching `toolResult` to validate ‚Äî then one **user** turn carrying one `toolResult` block per call is appended. The forcing pass (iteration cap hit) re-sends **without `toolConfig`** so the model must emit text.
- **`<thinking>` strip + image fixtures gotcha:** `extractText()` strips any `<thinking>‚Ä¶</thinking>` a model inlines into the visible reply (Nova leaks it; Claude doesn't) so raw reasoning never reaches Discord. Bedrock also rejects degenerate 1√ó1 images ("may not meet the required format"); real IG flyer covers are full-size and fine ‚Äî the smoke test synthesizes a real 64√ó64 PNG for the vision check.

### Image attachments (vision input, `src/attachments/`)

The bot accepts **images only** ‚Äî `resolveAttachments()` drops anything that isn't `png`/`jpeg`/`gif`/`webp` (detected by content-type first, then file extension) with a `Unsupported attachment type, skipping` warning. It is called in `handlers.ts` *before* `buildTurn()` and attaches the resolved images to the user `Turn`. Caps (config): `MAX_ATTACHMENT_COUNT` (5 ‚Äî extras silently ignored) and `MAX_ATTACHMENT_BYTES` (10 MB ‚Äî oversize skipped); downloads have a 30 s abort timeout and a failed download is logged, not fatal. `ImageAttachable` is provider-neutral (just `bytes`/`mimeType`/`format`); in `llm/client.ts`, `buildMessage()` emits a Converse content array (a `{ text }` block + one `{ image: { format, source: { bytes } } }` block per image) **when attachments are present**, else a single `{ text }` block. Attachments ride only the *current* user turn; history turns reconstructed by `buildHistory()` carry text only.

### Capabilities and tools

Each Capability composes one or more `ToolSource`s via `composeToolSources()` (`src/tools/source.ts`). Tool specs are **provider-neutral** (`{ name, description, inputSchema }`); the LLM client is the only place that knows how to wrap them into Bedrock Converse's `{ toolSpec: { name, description, inputSchema: { json } } }` shape. Tool name collisions across sources fail at boot ‚Äî fix the duplicate, don't suppress.

### Persistence

Single SQLite file, **one row per capability+version in `_migrations`** (see `src/memory/migrations.ts`). Each capability owns its tables by **id-prefix convention** (`calendar_events`, `instagram_monitor_accounts`, `configuration_bindings`) ‚Äî the framework does not enforce this, but the `configuration` capability's admin tools rely on it for scoped inspection.

`NamespacedMemory` wraps the shared store for one capability: it forwards `db()` to the raw handle but fixes `capabilityId` on `migrate()` so a capability cannot migrate another's tables. `__framework__` is reserved for framework-level state (currently just the user directory).

### Routing

`src/capabilities/routing.ts` builds a `MutableCapabilityRouter` from an initial channel‚Üícapability map. The `configuration` capability holds the only reference that can call `setBinding`/`removeBinding` ‚Äî read-only consumers type their dep as the parent `CapabilityRouter`. Bindings are persisted to SQLite, so live re-bindings from chat survive restarts; **no bot restart needed when re-binding a channel.**

### Calendar capability (global)

Rewritten 2026-06-21 from per-user to a single **global** server calendar. Mods talk to it in the bound **input** channel; rendered month PDFs + a master ICS are published to a separate **output** channel.

- **Two channels, two roles.** The capability is bound to the INPUT channel via the normal routing table (`config_bindings action:bind ‚Ä¶ capability:calendar`). The OUTPUT channel is a calendar-specific setting: seeded on first boot from `CALENDAR_OUTPUT_CHANNEL_ID` into `calendar_settings` (single row), then the DB value wins (change it from the config channel: `config_calendar action:set_output_channel`). Resolution order at runtime: DB setting ‚Üí `CALENDAR_OUTPUT_CHANNEL_ID` ‚Üí none (publish becomes a no-op reported as `published.ok=false`).
- **Global store** (`store.ts`, migration **v5**): `calendar_events` dropped per-user scoping (`created_by` is now attribution only ‚Äî any mod sees/edits any event). v5 also adds `calendar_published` (tracks the Discord message id per artifact so we EDIT in place) and `calendar_settings`. Recurrence (`daily`/`weekly`/`monthly` + optional `recurrence_until`) lives in `recurrence.ts`; `expandOccurrences` is reused by the snapshot, the renderer, and the ICS.
- **Recurring-event exceptions** (migration **v6**, `calendar_event_overrides`, added 2026-06-21): edit or cancel a SINGLE occurrence, or "this and following". A row overrides/cancels the occurrence whose ORIGINAL anchor time is `occurrence_start_at` (NULL field = inherit); `expandOccurrences` applies them (cancel ‚Üí skip, retime ‚Üí shifted, same-day only). The tools `calendar_update_event`/`calendar_delete_event` take a **`scope`**: `series` (default; also for one-offs), `occurrence` (just `occurrence_date_iso`), or `following`. "Following" is a **series split** (no override rows): the original master's `recurrence_until` is truncated to just before the split and a NEW master is created from the split point (delete-following just truncates). Splitting at the first occurrence collapses to a whole-series edit. A whole-series `start_at`/`recurrence_freq` change wipes the master's overrides (the rhythm changed). The prompt makes the model ASK "¬øsolo ese d√≠a, este y los siguientes, o toda la serie?" when scope is ambiguous, and the tool result carries `updated_scope`/`deleted_scope`.
- **Follow-up conversation** needs no new infra ‚Äî the existing reply-chain `buildHistory` already threads multi-turn context. The system prompt (`capability.ts`) drives it: require title + start date/time before creating, ask one concise question at a time for missing/ambiguous bits (location, "¬øse repite?"), Spanish-first.
- **PDF rendering** (`render.ts` + `template-geometry.ts`). The 7 templates in `calendar/` (Jun‚ÄìDec 2026, **not gitignored** ‚Üí they deploy with the repo and are read at runtime from the repo root) are Canva designs with the grid, day numbers and branding already drawn. `scripts/calibrate-calendar-templates.ts` bakes per-month grid geometry into `template-geometry.generated.ts` (poppler, **dev-time only**): **rows** come from the day-number text bounding boxes (`pdftotext -bbox`); **columns** come from detecting the true vertical grid lines in a grayscale raster (`pdftoppm -gray`), because the day numbers are right-aligned and inset ~8px from the real cell border ‚Äî using the text edge made event blocks bleed left across the gridline (fixed 2026-06-21). The horizontal grid is identical across templates, so the calibration applies a **consensus** column set to all 7 (one template's line-detection miss can't desync it). Row pitch differs (~97pt for 5-row months, ~80pt for the 6-row Aug/Nov). Each day's (row,col) is computed **arithmetically** (Monday-first, fixed UTC-6) by the pure grid math in `grid.ts`, which is shared by the renderer and the publisher (so both agree on placement). The renderer (pdf-lib, standard Helvetica) overlays compact red "event blocks" (bold time + title) into the cell; recurring events expand so a weekly series lands in every week and every month. Text is sanitized to WinAnsi (emoji/CJK stripped, smart-punctuation folded) so pdf-lib never throws. **Re-run the calibration script if the templates are ever replaced.**
- **PNG hero** (`raster.ts`): Discord shows a PDF as a non-previewing download card, so the publisher rasterizes each composed month PDF to a **PNG via `pdftoppm` at 150 DPI** (3000√ó1688, ~0.65 MB) and attaches **only the PNG** so Discord renders it inline (the PDF is dropped ‚Äî cleaner cards; re-add it to the payload if print-quality downloads are wanted again). This is a **runtime** poppler dependency (installed at `/usr/bin`, on the systemd PATH); if it's ever missing, raster throws and the publisher falls back to attaching the PDF so the month still posts (`calendar.raster_failed` logged once).
- **ICS** (`ics.ts`): one VEVENT per master with RRULE for recurring series (never pre-expanded), a fixed-offset `VTIMEZONE America/Mexico_City` (-0600), stable UIDs (`chopperbot-cal-<id>@revolucionz`) so re-importing updates rather than duplicates, RFC-5545 escaping + 75-octet line folding. Per-occurrence exceptions map to standard iCal: a cancelled occurrence ‚Üí `EXDATE` on the master VEVENT; a retimed occurrence ‚Üí a second VEVENT with the same UID + `RECURRENCE-ID` (original anchor) and the new DTSTART/SUMMARY. (A downloaded `.ics` is a one-time import snapshot ‚Äî there's no hosted webcal URL, so the copy refreshes on the single ICS message but members re-import to pick up changes.)
- **Publishing** (`publisher.ts`, `OutputChannelPublisher`): every mutation (and bot startup via `CalendarCapability.start()`) calls **`reconcile()`** ‚Äî a single idempotent op that makes the output channel match the DB. The **desired card set** is: every month containing a **one-off** event, plus the **current month** if it has any occurrence. Crucially, a **recurring series only shows on the current month's card ‚Äî it never spawns future-month cards** (this fixed the 2026-06-21 bug where one weekly event blasted all 7 months/8 messages). reconcile renders each desired month (PDF + PNG hero, concurrently), edits its tracked message in place (`calendar_published`), **deletes** any tracked card no longer desired (so the board self-cleans on rollover / when events are removed), and refreshes the single ICS message. Per month: one message = **PNG hero** (PDF only as a raster fallback). The ICS is a **single dedicated message**. Best-effort: failures never fail the DB mutation; the tool result carries a `published` summary (`posted`/`removed`) the model relays. `calendar_publish` just forces a `reconcile()` on demand. A one-off booked in a future month DOES get its own card; a recurring series in a future-only window shows on the current card once it reaches it (or appears in the ICS meanwhile).
- **Proof scripts** (not tests, write to `/tmp/chopper-cal`): `scripts/render-calendar-sample.ts` (renders the real example events + stress cases to PDF/PNG/ICS) and `scripts/calendar-e2e-proof.ts` (drives the real tool handlers against an on-disk DB, reopens it to prove persistence, renders the final state, validates the ICS). Verify renders visually with `pdftoppm -png -r 150 <pdf>`.
- **Wiring the test deployment:** set `CALENDAR_OUTPUT_CHANNEL_ID` in `.env` (done: the Revoluci√≥n Z output channel), then from the config channel bind the input channel to `calendar` (`config_bindings action:bind`). `pnpm run build && systemctl --user restart chopperbot.service` to pick up the env/output-channel change.

### Instagram monitor scheduler

`InstagramMonitorScheduler` ticks every minute (`DEFAULT_TICK_MS`), polling at most `ACCOUNTS_PER_TICK=1` due account per tick (`MAX_PUSHES_PER_ACCOUNT_PER_TICK_PER_CHANNEL=5`). Each account's poll interval is **adaptive** ‚Äî learned from its own posting cadence (see "Adaptive polling & budget governor" below), not a flat 60 min; `DEFAULT_POLL_INTERVAL_MS` (60 min) now survives only as the scheduler's nominal/log value (the per-account cold-start fallback is 12 h). **First-ever poll for an account seeds the dedup anchor and does NOT backfill** ‚Äî only posts strictly newer than the last seen IG post id are processed.

**One account per tick + jitter (anti-burst).** Polling is one account per minute (never a synchronized burst), and `store.dueAccounts()` adds a per-account jitter of up to `POLL_JITTER_FRACTION` (50%, now defined in `store.ts`) of **that account's own adaptive interval** to its next-due time, so accounts decorrelate and requests scatter irregularly across the window. This both looks less bot-like and reduces IP throttling.

**Anti-detection layers (see scheduler.ts / fetcher.ts).** Designed to survive IG automation detection on a *personal* account (a banned account is catastrophic):
- **Quiet hours** 02:00‚Äì07:00 America/Mexico_City, with a deterministic per-day ¬±20-min jitter on each edge (`inQuietHours`) so the resume time isn't a fixed daily tell.
- **~8% random whole-tick skips** (`tickSkipProbability`, set in `capability.ts`) so polling isn't a perfect metronome.
- **Per-poll randomness in `fetcher.ts`**: 50%-probability HTML warmup + 1‚Äì3 s delay, a 400‚Äì1500 ms human delay between the pk-resolve and the feed request, and a randomized feed `count` (10‚Äì16).
- **`IG_USER_AGENT`** is sent on every API *and* CDN request (`publisher.ts` shares it via `setIgCdnUserAgent`) ‚Äî set it to **match the browser the cookies were extracted from**; a mismatched UA is a fingerprint signal.

**Adaptive polling & budget governor (store migrations v6 + v7, added 2026-05-29).** A flat 60-min cadence starved the 120-request budget across many accounts, round-robining it equally so a 6-day-cadence account got the same share as a 2-hour one. Now each account's interval is **learned from its own `posted_at` history** and the budget is spent where posts actually happen. Detection *latency* varies, not coverage (the feed returns 10‚Äì16 items, so a longer interval can't miss posts unless an account posts >~10√ó within it ‚Äî only the most active accounts, which get the shortest intervals).
- `computeCadenceInterval()` (in `store.ts`, pure + unit-tested) takes an account's distinct recent post times (**deduped across channels** ‚Äî `seen_posts` is per-channel, so the query `GROUP BY ig_post_id` or the median halves) and returns `interval = clamp(median_inter-post_gap √ó 0.5, 45 min, 12 h)`, with a recency-decay stretch toward MAX once an account goes quiet, and a coverage clamp. (The ceiling was raised 6 h ‚Üí 12 h on 2026-05-30: live data showed the 6 h cap was clamping genuinely-rare accounts ‚Äî e.g. `semillasderebeldia` ~20 h median gap wants ~10 h, `revueltasperiodico` ~26 h wants ~13 h ‚Äî to roughly half their natural interval, wasting the request budget the active accounts need. Per-account interval is purely a budget lever; IG rate-limits on the *session's* aggregate rate, not per-account, so a longer ceiling carries no extra ban risk.) It returns `null` (‚Üí cold-start) until there are ‚â• `CADENCE_MIN_SAMPLES` (5) posts spanning ‚â• `CADENCE_MIN_SPAN_MS` (**24 h**) ‚Äî `MIN_SAMPLES` rejects tiny bursts; the 24 h span rejects a single-afternoon flurry while still accepting a real multi-day cadence (a 26-post/2.9-day account would be wrongly rejected at a 3-day threshold).
- **Cold start** (`CADENCE_COLD_START_INTERVAL_MS` = 12 h, kept == MAX): accounts without a trusted cadence poll slowly *on purpose* ‚Äî few posts ‚áí empirically a rare poster, so we don't burn budget on unknowns; they speed up automatically as history accrues. (This is why the default fallback is 12 h, not the old 60 min. Polling slowly does NOT slow learning ‚Äî cadence is learned from detected posts, and a 12 h interval can't miss a rare account's posts since the feed window holds 10‚Äì16.)
- The interval is cached on `instagram_monitor_accounts` (`poll_interval_ms`, `posts_per_day`, `cadence_updated_at`; **v6**) and read by `dueAccounts()` via `effectiveBaseIntervalMs()` = `(poll_interval_ms ?? cold-start) √ó poll_stretch`, clamped to MAX. The failure-backoff cap is decoupled to `min(base √ó 2^failures, max(MAX_BACKOFF_MS, base))` ‚Äî `MAX_BACKOFF_MS` stays 6 h while the cadence ceiling is now 12 h, so this `max(‚Ä¶)` guard genuinely binds: a 12 h-cadence account with failures still polls every ~12 h, not exponentially slower.
- **Budget governor** (`computeGovernorStretch()`): a single global `poll_stretch` (runtime row, **v7**) raised above 1.0 so the **realized** projected daily requests stay ‚â§ `IG_DAILY_REQUEST_BUDGET √ó CADENCE_BUDGET_HEADROOM` (**0.75**). It is **clamp-aware** (fixed 2026-05-30): because `interval √ó stretch` saturates at the 12 h ceiling, an already-maxed account contributes a fixed request rate the stretch can't reduce ‚Äî so the old `projected / ceiling` closed form under-corrected whenever accounts sat at the clamp, letting realized spend overshoot. The governor now binary-searches the smallest `stretch` whose *realized* (post-clamp) projection `Œ£ expectedPollsPerDay(min(interval √ó stretch, MAX)) √ó callsPerPoll` meets the ceiling; one multiplier preserves the active-vs-rare allocation, and the hard budget gate stays the backstop. It is also **quiet-aware per account** (fixed 2026-06-12, was residual #1): `expectedPollsPerDay()` in `store.ts` models each account as `min(DAY/spacing, (DAY‚àíquietWindow)/spacing + 1)` ‚Äî quiet hours only *time-shift* a long-interval account's polls (a 12 h-ceiling account polls ~2√ó/day regardless), so the old uniform `activeFraction` (‚âà0.73) under-projected every clamped account. With 7 of 18 accounts clamped, the model said 60 polls/day while the journal showed 70 (+17%), and realized spend sat at **106/120** ‚Äî only ~14 requests of margin, which a restart's pk-resolve burst (~+18 calls) would have blown through. `spacing` also includes the correct jitter expectation: `pollJitterMs` re-rolls Uniform[0, interval/2) **every tick**, so the expected extra delay is the first-passage time ‚âà `sqrt(œÄ/2 √ó J √ó tick)` (~8 % at 2 h, ~3 % at 12 h), NOT J/2. The random tick-skip is deliberately *not* in the model (a skipped tick delays a due poll by one tick; it never drops one). `callsPerPoll` is **measured, not guessed** (changed 2026-05-31): the scheduler keeps a rolling-24 h count of *polls* (`pollTimestamps`, mirroring `requestTimestamps`) and feeds the governor `requests √∑ polls`, clamped to `[1, 3]`, falling back to `CALLS_PER_POLL_FALLBACK` (**1.7**) until `CALLS_PER_POLL_MIN_SAMPLES` (20) polls accrue. The old hardcoded **1.5** systematically *under-projected* ‚Äî live realized is **~1.7** (warmup fires a hair over 50 %, pk occasionally re-resolves), so the governor set too low a stretch and realized spend overshot the headroom target (112/120 instead of the intended ~90). The self-calibrating value also tracks any future drift in warmup/pk behavior, and is surfaced as `calls_per_poll` on the `instagram_monitor.cadence_sweep` log line. The 25 % headroom now only absorbs the restart pk-resolve burst (~+1 call/account in-window for 24 h ‚Äî ~18 calls at the 2026-06-12 fleet size) + `callsPerPoll` measurement lag: the polls-projection gap that used to eat it (realized ~65‚Äì70/day vs modeled ~60) was closed by the quiet-aware `expectedPollsPerDay` model (2026-06-12, below).
- **Recompute triggers:** (1) the per-account *cadence interval* is recomputed opportunistically right after a poll detects new posts (single account, in `processAccount`) and in a once-per-`CADENCE_TTL_MS` (24 h) sweep over all accounts (`recomputeAllCadence`); (2) the global `poll_stretch` is recomputed **every tick** from the *current* cached intervals (`recomputeGovernorStretch`, decoupled from the 24 h sweep on 2026-06-02 ‚Äî see below). Both run from the tick's `finally` block, **even while polling is paused** (pure SQLite, no IG calls) ‚Äî recomputing cadence + stretch is exactly what lets a budget-pinned monitor shrink its intervals and recover. Log lines: `instagram_monitor.cadence_sweep` (daily: `swept` + `stretch` + `projected` + `calls_per_poll`) and `instagram_monitor.stretch_update` (per-tick, emitted only on a ‚â•10 % stretch change to avoid journal spam).
- **The stretch must be recomputed continuously, not once per day (fixed 2026-06-02).** Originally `poll_stretch` was refreshed only in the 24 h cadence sweep. But per-account intervals shrink *throughout the day* ‚Äî active accounts tighten as they post (opportunistic recompute) ‚Äî and the sweep is locked to whatever wall-clock hour the bot last restarted. On a Pi that restarted at noon, the sweep snapshotted stretch at the daily *activity low* (overnight quiet + morning lull), then the afternoon/evening posting surge tightened intervals and realized spend climbed straight to the **120/120 hard cap** (observed 2026-06-02: noon sweep set `stretch=2.087`/`projected=90`, but the live-correct stretch by 15:00 was ‚âà**2.67**; realized hit 120 and soft-paused ~20 min repeatedly from ~14:00 on). Per-account *cadence* recompute can stay daily (rare-account cadences move slowly; active ones get the opportunistic refresh), but the *governor stretch* ‚Äî a pure function of the current cached intervals + measured `callsPerPoll` ‚Äî is now recomputed every tick so it tracks the intra-day activity cycle. The hard budget gate stays the backstop; this just keeps the monitor from leaning on it.

Net effect on the real 14-account deployment, with a correction logged after a full day of live data:
- **Per-account adaptation works as designed.** A day after the 12 h ceiling shipped, the rare accounts had converged to their true intervals (`revueltasperiodico` ‚Üí 12 h, `semillasderebeldia` ‚Üí ~9.4 h, `gentrificacionentuidioma` ‚Üí ~7.6 h, `revolucionzmexico` cold-start ‚Üí ~9.8 h) and accounts that turned out *more* active than the snapshot tightened (`un_altavoz` cold-start ‚Üí 81 min once it showed ~5.5 posts/day, `antigentrificacion.cdmx` 120 ‚Üí 51 min). All 14 with zero failures.
- **But the predicted budget convergence did NOT materialize the way the 2026-05-30 snapshot implied.** That snapshot predicted `poll_stretch` ‚âà2.0 ‚Üí **‚âà1.1** and realized spend ‚Üí **~90 calls/day**. In reality, a day later `poll_stretch` settled at **~2.1** (it *rose*) and realized spend was **~112/120** ‚Äî because (a) overall account activity rose above the static snapshot (the budget freed by rare accounts was immediately consumed by the newly-active ones), and (b) the governor's hardcoded `callsPerPoll = 1.5` under-counted the real ~1.7, so it set too low a stretch. Both push realized up. The system never tripped `budget_exhausted` (the 25 % headroom absorbed the overshoot) but ran with only ~8 requests of margin to the hard cap instead of the intended ~30.
- **The 2026-05-31 fix** (self-calibrating `callsPerPoll`, above) closes (b): the governor now projects against the measured ~1.7, raises stretch enough to pull realized back toward the headroom target, and restores real margin to the hard cap. (a) is not a bug ‚Äî it's the model correctly spending the budget where the activity is; it just means `poll_stretch` is a *dynamic* function of the live account mix, not a number that settles at a documented constant.
- **2026-06-12 (quiet-aware `expectedPollsPerDay`) ‚Äî validation pending.** At deploy (18 accounts, 7 ceiling-clamped) the governor jumped stretch 2.46 ‚Üí **3.48** with `calls_per_poll` at the 1.7 fallback (restart wiped the measurement window; it should relax toward ~1.5 within hours as 20+ polls accrue, letting stretch ease too). **Check after ‚â•24 h uptime:** `requests_24h` should read **~85‚Äì95** (was 106) and realized polls/day should match the model within ~¬±5 % (see the polls-bias check below). If realized spend still sits >100 with the journal showing no model bias, the gap is `callsPerPoll` lag (residual #2) or fleet growth ‚Äî NOT a reason to re-add an `activeFraction`-style fudge. If the polls-bias check itself shows >10 % bias again, something new is unmodeled: measure first, then extend `expectedPollsPerDay`, don't tune constants blind.

The takeaway for future readers: **trust the live DB + `cadence_sweep` log over any hardcoded "expected" stretch/spend number in this doc** ‚Äî those are snapshots of a moving target. The invariants that *do* hold: per-account intervals track each account's own cadence (clamped to [45 min, 12 h]), and the governor holds *realized* spend under `0.75 √ó budget` plus a now-accurate headroom.

**Observing the cadence/governor (do this before changing anything).** The whole system is a feedback loop over live data, so *measure first*. The exact research workflow:
- **Realized spend & current stretch:** `sqlite3 data/chopperbot.db 'SELECT poll_stretch, requests_24h, global_stop, budget_pause_until FROM instagram_monitor_runtime;'`. `requests_24h` is the rolling-24 h IG-call count; it should sit **~90‚Äì100/day** (headroom target 90, +residual) and should **not** repeatedly tap `IG_DAILY_REQUEST_BUDGET` (120). Note this resets to 0 on every restart (in-memory window) ‚Äî read it after the bot has been up ‚â•24 h for a true figure.
- **Per-account cadence vs reality:** compare each account's *cached* interval to its *actual* posting rate. Cached: `SELECT username, round(poll_interval_ms/60000.0,1) AS interval_min, round(posts_per_day,2) ppd, paused FROM instagram_monitor_accounts ORDER BY poll_interval_ms;`. Actual (last 60 d, deduped across channels): `WITH p AS (SELECT account_username u, ig_post_id, MAX(posted_at) t FROM instagram_monitor_seen_posts WHERE posted_at>=(strftime('%s','now')*1000-60*86400000) GROUP BY 1,2) SELECT u, count(*) n, round(count(*)*1.0/NULLIF((MAX(t)-MIN(t))/86400000.0,0),2) ppd FROM p GROUP BY u ORDER BY ppd DESC;`. Cached `ppd` should ‚âà actual `ppd`; a large gap means the cadence estimate is stale or the trust gate is rejecting real history.
- **Intra-day activity curve** (why the once-daily snapshot failed): `SELECT strftime('%H:00',detected_at/1000,'unixepoch','localtime') h, count(DISTINCT ig_post_id) n FROM instagram_monitor_seen_posts WHERE detected_at>=(strftime('%s','now')*1000-36*3600000) GROUP BY 1;`. Expect a quiet morning and an afternoon/evening surge ‚Äî the stretch should rise/fall with it (watch `instagram_monitor.stretch_update`).
- **Governor trail in the journal:** `journalctl --user -u chopperbot -o cat | grep -E 'cadence_sweep|stretch_update|budget_exhausted'`. `cadence_sweep` is the daily per-account recompute (carries `stretch`/`projected`/`calls_per_poll`); `stretch_update` is the per-tick governor move (‚â•10 % changes only). `projected` is always ‚âà the headroom ceiling **at the moment it logged** ‚Äî the value is correct, what used to go stale was *how often* it was recomputed.
- **Polls-bias check (the diagnostic that found the 2026-06-12 fix).** Compare *realized* polls/day against what the governor *modeled*: realized = `journalctl --user -u chopperbot --since "-24 hours" -o cat --no-pager | grep -c '"msg":"instagram_monitor.fetch.ok"'`; modeled = `projected √∑ calls_per_poll` from the latest `stretch_update`/`cadence_sweep` line. Because the stretch is re-solved every tick to hold the instantaneous projection at the headroom ceiling, **any persistent realized-vs-modeled gap is pure model bias** (intra-day drift can't explain it). Pre-fix this read 70 vs 60 (+17 %); healthy is within ¬±5 %. Cross-check spend the same way: `requests_24h √∑ realized polls` should ‚âà the logged `calls_per_poll` ‚Äî if it does, the poll *count* model is what's off, not the per-poll cost (that's exactly how the 2026-06-12 diagnosis isolated the quiet-hours term).

**Known residuals / open follow-ups (not yet fixed, ordered by impact ‚Äî revisit if `requests_24h` still creeps toward 120):**
1. ~~`activeFraction` under-counts ceiling-clamped accounts~~ ‚Äî **FIXED 2026-06-12** with the quiet-aware `expectedPollsPerDay()` (see the budget-governor bullet above). The 9-day live validation that motivated it: realized 70 polls/day vs modeled 60; the quiet-aware model predicts ~72 against the same snapshot (-3 % error instead of +17 %). Expect realized spend to settle back toward ~90/day and `poll_stretch` to RISE (‚âà2.5 ‚Üí ‚âà3.3 at the 2026-06-12 account mix) ‚Äî that's the fix working, not a regression.
2. **`callsPerPoll` measurement lag.** It's a trailing-24 h `requests √∑ polls` ratio used to project the *next* 24 h, so it lags real drift (measured ~1.45‚Äì1.5 in the 2026-06-12 window vs ~1.7 in late May ‚Äî it genuinely moves). The per-tick recompute (2026-06-02) re-reads it every minute, which mitigates this, but it's still a backward-looking estimate.
3. **Account count drift:** prose counts go stale fast (14 ‚Üí 15 ‚Üí **18** as of 2026-06-12, 2 cold-start). Don't trust the count in prose ‚Äî `SELECT count(*) FROM instagram_monitor_accounts;`.

**Guardrails / circuit breaker ‚Äî the bot STOPS instead of escalating into a ban.** State lives in the `instagram_monitor_runtime` table (store migration **v5**, a single row id=1) so it survives restarts; `clearFailureBackoff()` deliberately never touches it.
- **429 / throttle** ‚Üí `InstagramRateLimitError` ‚Üí global in-memory cooldown (base 2 h, √ó2 per throttle in a 6 h window, cap 12 h); halts ALL polling. Two throttles in 6 h trips the persistent kill-switch.
- **Persistent kill-switch (`global_stop`)** ‚Äî manual-resume-only. Trips on a SESSION-level auth failure (`require_login`/checkpoint/challenge ‚Äî see below) or repeated throttles, fires `postCircuitBrokenAlert` to the config channel, and **stays stopped across restarts** until an operator runs `config_instagram action:resume_monitor confirm:true`. `tickOnce()` reads the row first and returns early while stopped (`instagram_monitor.tick.global_stop`).
- **Session vs account-specific auth failures.** `InstagramAuthError.sessionLevel` is true only for `require_login`/checkpoint/challenge/consent (the cookies/account are flagged ‚Üí halt everything + trip). A **bare 401/403 on one handle** (private/restricted/blocked-relative-to-your-account, or transient) is logged `instagram_monitor.account_auth_failed` and only bumps that account's `consecutive_auth_failures` ‚Üí it auto-pauses at `AUTH_PAUSE_THRESHOLD` (5) via `dueAccounts()` while the other accounts keep polling. **Do not** treat a per-account 401 as session death ‚Äî that would stop the whole monitor over one bad handle.
- **Daily request budget** `IG_DAILY_REQUEST_BUDGET` (default 120) ‚Äî a rolling-24 h ceiling on outbound IG HTTP calls (the scheduler counts them via `fetcher.observeRequests`); on hit, polling soft-pauses (auto-recovers as the window drains) + one alert (`postBudgetExhaustedAlert`), and `postResumedAlert` fires once on recovery. One poll ‚âà **~1.7 calls steady-state** (feed + ~50 % warmup + occasional pk re-resolve; pk is otherwise cached, so the first poll of each account after a restart is the only reliably ~2‚Äì3-call one). This figure is **measured live**, not assumed ‚Äî see the `callsPerPoll` discussion under the budget governor. The adaptive **budget governor** (above) keeps the steady-state projection below this ceiling, so it should rarely trip.
- **Admin visibility**: `config_instagram action:status` reports kill-switch state/reason, active cooldowns, 24 h request count, paused/auth-blocked counts, an `adaptive` block (accounts with a learned cadence + the governor `poll_stretch`), and a `digest_preview` (the exact daily-digest text). `action:digest_now` returns that digest on demand; `action:list` now includes each account's `poll_interval_min` / `posts_per_day`. The scheduler mirrors its in-memory cooldowns into the runtime row once per tick (`writeHeartbeat`) so the (DB-only) admin tool can see them.

**Pinned-post ordering gotcha.** Instagram returns *pinned* posts first, out of chronological order. The scheduler sorts fetched posts by `takenAtMs` before the dedup walk and anchor advance ‚Äî otherwise a pinned (old) post sitting at array index 0 would freeze detection (the walk would hit the anchor immediately and never see newer posts below the pins).

**Anchor-missing gotcha (the dedup anchor is `(id, takenAtMs)`, not just an id).** The walk down to the anchor id assumes the anchor post is still in the returned window. IG sometimes returns a *stale or paginated* window that **omits** the anchor post (eventual consistency, or the post was deleted). The old code then treated the entire returned batch as "new" and **backfilled weeks-old posts**, and `markPollSuccess` moved the anchor *backward* to that batch's older newest ‚Äî re-arming the backfill every poll. The fix: each account also stores `last_post_at` (the anchor's `takenAtMs`, added in store migration v3, backfilled from `MAX(seen_posts.posted_at)`). When the anchor id is present we still slice above it (preserving pinned handling); when it's **absent** we fall back to a strict capture-time gate (`takenAtMs > last_post_at`), and the anchor is only ever advanced *forward* in time. Watch for `instagram_monitor.anchor_missing.time_gated` (a stale window was rejected) and `‚Ä¶anchor_missing.reseed` (a legacy row with no recorded `last_post_at` re-seeding once).

**Accounts are global** (one row per username, `instagram_monitor_accounts.UNIQUE(username)`). On every detection cycle the scheduler reads the live router for the list of channels currently bound to `instagram_monitor` and **fans out each new post to every bound channel** with per-channel dedup via `instagram_monitor_seen_posts (ig_post_id, channel_id)`. Consequences:
- A newly-bound channel sees no backfill ‚Äî its `seen_posts` rows start empty and the global `last_post_id` anchor is already current.
- If no channels are bound when a poll fires, the anchor still advances; no posts are classified or recorded as seen.
- Pause / remove / force-poll affect every bound channel.

Posts are pushed in chronological order even though the IG endpoint returns newest-first.

The bot fetches **direct** from the host's residential IP ‚Äî there is no longer any fetch indirection. (A former `INSTAGRAM_RELAY_LAMBDA_ARN` AWS Lambda relay was removed on 2026-05-30: Instagram blocks all AWS datacenter IP ranges, so rotating within the Lambda pool never helped, and it was unused.) Direct mode requires the spoofed `sec-fetch-*` headers in `fetcher.ts` because Node's undici sends `sec-fetch-site: cross-site` by default and IG rejects it with a 400.

**Two fetch modes in `DirectInstagramFetcher`:**
- **Anonymous** (no `IG_SESSIONID`): hits the public `web_profile_info` GraphQL endpoint. Returns timeline media but is heavily IP-throttled (frequent `401 require_login`).
- **Authenticated** (`IG_SESSIONID` + `IG_CSRFTOKEN` + `IG_DS_USER_ID` set, optional `IG_MID`/`IG_DID`): far higher rate limits. **Gotcha:** authed `web_profile_info` returns an *empty* timeline, so auth mode does a two-step fetch ‚Äî resolve the account's numeric pk via `web_profile_info` (cached per handle), then read the private `feed/user/{pk}` endpoint (different REST item shape, parsed by `parseUserFeedBody`). The bare post id is recovered from the string `item.id` (`"{pk}_{owner}"`), not the numeric `item.pk` (which exceeds `Number.MAX_SAFE_INTEGER` and loses precision); this keeps dedup anchors consistent with the old GraphQL ids. The deployment now uses a **personal** IG account (the throwaway was flagged after ~4 days), which is why the guardrails above (hard stop on real signals, UA matching the cookie source, low/irregular volume) exist ‚Äî automated polling always carries some ban risk.

**Session expiry ‚Üí Discord alert.** A SESSION-level auth failure (`require_login` body, or checkpoint/challenge) throws `InstagramAuthError` with `sessionLevel=true`; the scheduler logs `instagram_monitor.auth.expired`, posts `postAuthExpiredAlert` to the config channel (rate-limited 1/6 h), and trips the kill-switch. To recover: clear any challenge in the IG app, refresh `IG_SESSIONID`/`IG_CSRFTOKEN`/`IG_DS_USER_ID` (+`IG_MID`/`IG_DID`) in `.env` from the **same browser** as `IG_USER_AGENT`, `pnpm run build && systemctl --user restart chopperbot.service`, then `config_instagram action:resume_monitor confirm:true`. (The old macOS launchd watcher that turned the log line into a Sosumi notification does not exist on the Pi ‚Äî the bot now alerts Discord directly.)

**Resume alert + daily status digest (config channel).** Two operator-facing signals sit alongside the pause alerts:
- `postResumedAlert` (`‚úÖ sondeo REANUDADO`) fires once when polling resumes after an **abnormal** pause ‚Äî budget window drained, rate-limit cooldown elapsed, or the kill-switch cleared. The scheduler tracks one `lastBlockReason` set at each abnormal gate (NOT quiet-hours/random-skip, which are normal and leave it untouched) and emits at the proceed-point, debounced (`MIN_PAUSE_FOR_RESUME_MS` 10 min + `RESUME_ALERT_COOLDOWN_MS` 30 min) so a flapping budget can't spam. Seeded from `global_stop` in `start()` so a kill-switch cleared after a restart still alerts once. Log: `instagram_monitor.tick.resumed`.
- A **daily status digest** posts at `STATUS_DIGEST_HOUR` (21:00 America/Mexico_City) from the tick's `finally` block (so it runs regardless of the gates above), once per local day (`instagram_monitor_runtime.last_digest_at`, stamped *after* a successful send so a failed send ‚Äî e.g. a restart landing in the digest hour before Discord is reachable ‚Äî retries next tick). `formatStatusDigest()` (`format.ts`, shared with the admin `status`/`digest_now` so they can't drift) renders overall state, 24 h budget/headroom, governor stretch, and a per-account table (cadence, effective interval, last-post age, next-poll ETA, flags), truncated under Discord's 2000-char limit. This revives the role of the decommissioned macOS 21:00 daily-summary.

## Env & configuration gotchas

- **dotenv `override: false`.** A stale `export ACCESS_KEY_ID=...` (or any other config var) in `~/.zshrc` shadows `.env`. The fix is `unset` in your shell, not flipping override ‚Äî legitimate dev workflows depend on shell-var overrides.
- **Required** at boot: `DISCORD_TOKEN`, `CHOPPERBOT_CONFIG_CHANNEL_ID`, `ACCESS_KEY_ID`, `SECRET_ACCESS_KEY`. Anything else has a default in the Zod schema (`AWS_REGION` ‚Üí us-east-1, `BEDROCK_MODEL_ID` ‚Üí `us.anthropic.claude-sonnet-4-6`).
- **AWS Bedrock is the LLM backend** (migrated off Kimi 2026-06-23). The bot calls the Bedrock Converse API with the IAM key pair `ACCESS_KEY_ID`/`SECRET_ACCESS_KEY` (+ optional `AWS_SESSION_TOKEN`) in region `AWS_REGION`. There is **no** other AWS integration ‚Äî no Secrets Manager, no Lambda relay (both long removed). The deployment's IAM user `chopperbot-readonly` was granted an inline `chopperbot-bedrock-invoke` policy (`bedrock:InvokeModel` on `amazon.nova-*` + `anthropic.claude-*` foundation models and their `us.`/`global.` inference profiles). The `AWS_DIVE_*` keys in `.env` are a **different** principal and are not used by the bot. A stray unknown `AWS_*` var is harmless (Zod ignores unknown keys).
- Three ways to seed channel‚Üícapability bindings via env (priority order, set in `src/config.ts`):
  1. `DISCORD_CHANNEL_CAPABILITIES` ‚Äî preferred. JSON array of `{ guildId?, guildName?, channels: [{ id, capability }] }`.
  2. `DISCORD_AUTHORIZED_CHANNELS` ‚Äî legacy: all listed channels run `DEFAULT_CAPABILITY`.
  3. `DISCORD_CHANNEL_ID` ‚Äî legacy single channel.
  After first boot these only seed; the source of truth is the DB, managed live via the `configuration` capability.

## Logs & observability

The bot logs JSON (pino) to stdout, which systemd routes to **journald**. There are no log files on the Pi.

```bash
journalctl --user -u chopperbot -f                    # live tail
journalctl --user -u chopperbot -f -o cat | npx pino-pretty   # pretty
journalctl --user -u chopperbot -b -n 100 --no-pager  # since boot / last 100 lines
```

Log lines worth recognizing: `Discord client ready` (gateway up), `InstagramMonitorCapability scheduler started` then recurring `instagram_monitor.tick` (poller alive), `instagram_monitor.auth.expired` (**IG cookies expired** ‚Äî refresh `IG_SESSIONID`/`IG_CSRFTOKEN`/`IG_DS_USER_ID` in `.env`, then `systemctl --user restart chopperbot.service`), `llm.health.alerting`/`llm.health.recovered` (Bedrock requests failing/recovered ‚Äî mirrors the Discord alert), `lifecycle.unclean_restart_detected` (previous process crashed), and pino `level: 50/60` = warn/error/fatal.

For ad-hoc state (no status UI on the Pi), query SQLite directly, e.g. `sqlite3 data/chopperbot.db 'SELECT username, consecutive_failures, last_polled_at FROM instagram_monitor_accounts ORDER BY username;'`.

If `systemctl --user` over SSH says "Failed to connect to bus", the session has no user D-Bus: `export XDG_RUNTIME_DIR=/run/user/$(id -u)`.

For local dev (not the deployed service), `pnpm run start | pino-pretty` still works.
